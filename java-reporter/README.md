# ResideoNextGen Reporter (Java)

A Java/Maven integration that lets **any** TestNG or Cucumber project
generate and view ResideoNextGen Dashboard data locally, with no Node/npm
install required on the consuming machine. Allure is still the single
source of truth — this module reads the Allure results your project's
existing Allure adapter already writes, and never captures or calculates
results on its own.

```
Your test run (TestNG or Cucumber)
        ↓
Allure adapter (allure-testng / allure-cucumber7-jvm — you already have this)
        ↓
target/allure-results/
        ↓
resideonextgen-reporter-core  (read → normalize → merge history → write JSON)
        ↓
target/resideonextgen/data/*.json
        ↓
embedded HTTP server (bundled React dashboard + live data)
```

## Modules

- **`resideonextgen-reporter-core`** — canonical model, Allure-results
  reader, the single calculation layer, history merge, static JSON writer,
  an embedded zero-dependency HTTP server (JDK's `HttpServer`, no Jetty/
  Tomcat), and thin TestNG/Cucumber lifecycle hooks. Add as a test-scope
  dependency.
- **`resideonextgen-maven-plugin`** — provides `mvn resideonextgen:serve`.
  Self-sufficient: processes whatever Allure results already exist and
  serves them, without needing the listener/plugin wired in at all.

## Why this exists

The core dashboard (`../`) is a static site generated by a Node/TypeScript
processor and deployed to GitHub Pages — great for CI/CD, but it means a
developer has to `npm install` and run a separate pipeline just to preview
results locally. This Java integration is for that local/inner-loop case:
add one dependency, and `mvn resideonextgen:serve` shows the same
dashboard, fed by the same kind of Allure data, with zero Node involved.

## Testing

### 1. Fast path: run the module's own tests (no Selenium, no browser)

```bash
cd java-reporter
mvn test
```

This runs three test classes in `resideonextgen-reporter-core`:

- `NormalizerTest` — exact totals/pass-rate math, and that
  `collapseRetries` correctly supersedes an earlier failed attempt with a
  later passed rerun instead of double-counting it.
- `AllureResultReaderTest` — parses a hand-written `*-result.json`,
  confirms `environment.properties` reads case-insensitively (Title-Case
  and UPPERCASE keys both work), and confirms the `suite` grouping key is
  read correctly.
- `ReportEngineIntegrationTest` — the real scenario end-to-end against a
  temp `allure-results` directory: a first partial run (2 passed, 1
  failed), a second partial run adding a new test *and* a rerun of the
  earlier failure that now passes, asserting it all merges into **one**
  report entry with correct collapsed totals; then a new `suite` value
  (simulating a fresh `mvn clean test` regression cycle) asserting a
  **second**, independent report entry appears.

This is the fastest way to verify the two newest, least-obvious behaviors
(retry-collapsing and suite-grouping) are correct, and requires nothing
beyond a JDK and Maven.

### 2. End-to-end: real Allure results through the real dashboard

Using the [sample project](../../saucedemo-selenium-sample), already wired
to this module:

```bash
cd java-reporter && mvn clean install
cd ../saucedemo-selenium-sample
mvn test
mvn resideonextgen:serve
```

Open the printed `http://localhost:<port>/` URL and check: the Overview
page loads, KPI totals match what `mvn test` printed in its Surefire
summary, and the donut/trend charts render. Stop with Ctrl+C.

To specifically exercise suite-grouping end-to-end: add `suite=<name>` to
whatever writes `environment.properties` in that project (see
`AllureEnvironmentWriter`), run `mvn test -Dtest=SomeSubsetOfScenarios`
twice in a row (no `mvn clean` between them) with the same suite name, then
`mvn resideonextgen:serve` and confirm the dashboard shows one execution
with the combined results, not two.

### 3. Optional: auto-serve smoke test

```bash
mvn test -Dresideonextgen.autoServe=true
```

Confirm the console prints a `ResideoNextGen Dashboard is live at ...` URL
right after the suite finishes, and that Ctrl+C cleanly stops it. Then
confirm a plain `mvn test` (no flag) does **not** start a server and exits
normally — this is the behavior a CI pipeline depends on.

## Quick start

### 1. Build and install locally

Not yet published to Maven Central — build once and install to your local
`~/.m2` repository:

```bash
cd java-reporter
mvn clean install
```

### 2. Add the dependency to your test project

```xml
<dependency>
  <groupId>com.resideo.nextgen</groupId>
  <artifactId>resideonextgen-reporter-core</artifactId>
  <version>1.0.0-SNAPSHOT</version>
  <scope>test</scope>
</dependency>
```

Your project must already have an Allure adapter configured for your
framework (this is normally 1-2 lines you likely already have):

```xml
<dependency>
  <groupId>io.qameta.allure</groupId>
  <artifactId>allure-testng</artifactId>       <!-- for TestNG -->
  <!-- or -->
  <artifactId>allure-cucumber7-jvm</artifactId> <!-- for Cucumber -->
  <scope>test</scope>
</dependency>
```

### 3. Run your tests, then view the dashboard

```bash
mvn test
mvn resideonextgen:serve
```

`mvn resideonextgen:serve` processes `target/allure-results` into
`target/resideonextgen/data`, starts a local server on a free port, and
prints the URL. Press Ctrl+C to stop. It's fully self-sufficient — you
don't need to add the listener/plugin below just to use this command.

If the `resideonextgen` goal prefix doesn't resolve, either declare the
plugin (no execution needed) in your `<build><plugins>`:

```xml
<plugin>
  <groupId>com.resideo.nextgen</groupId>
  <artifactId>resideonextgen-maven-plugin</artifactId>
  <version>1.0.0-SNAPSHOT</version>
</plugin>
```

or add `com.resideo.nextgen` to `<pluginGroups>` in `~/.m2/settings.xml`
once, or invoke it fully-qualified:
`mvn com.resideo.nextgen:resideonextgen-maven-plugin:1.0.0-SNAPSHOT:serve`.

## Optional: auto-serve immediately after tests finish

Off by default — a CI build must never hang waiting for someone to Ctrl+C
a server it never asked for. To opt in for local runs:

```bash
mvn test -Dresideonextgen.autoServe=true
```

This works automatically for **TestNG** suites: `resideonextgen-reporter-core`
registers `ResideoNextGenTestNgListener` via TestNG's ServiceLoader
mechanism (`META-INF/services/org.testng.ITestNGListener`), so it attaches
with zero configuration. When the suite finishes, it processes
`target/allure-results` and — only if the flag above is set — starts the
server and blocks (similar to `mvn spring-boot:run`) until you Ctrl+C.

For **pure Cucumber** runs (JUnit Platform, no TestNG suite wrapping),
register the plugin explicitly — either via `@CucumberOptions`:

```java
@CucumberOptions(plugin = {
    "pretty",
    "io.qameta.allure.cucumber7jvm.AllureCucumber7Jvm",
    "com.resideo.nextgen.reporter.cucumber.ResideoNextGenCucumberPlugin"
})
```

or (no code change) via `cucumber.properties` / `junit-platform.properties`:

```properties
cucumber.plugin=com.resideo.nextgen.reporter.cucumber.ResideoNextGenCucumberPlugin
```

If your Cucumber scenarios run through `AbstractTestNGCucumberTests` (as in
the [sample project](../../saucedemo-selenium-sample)), the TestNG listener
already covers you — Cucumber there is just the step-runner, TestNG is
still the suite that finishes and triggers processing.

## Grouping multiple runs into one report ("suite" grouping)

By default, every processing run gets its own execution entry (keyed by
the Allure executor's `buildOrder`, or a timestamp). For local/incremental
work, that's often not what you want — e.g. you run a subset of tests, run
another subset, then rerun a couple of failures, and want it all to land
in **one** report instead of three fragmented ones.

Set a `suite` value in Allure's `environment.properties` (the same file
your Allure adapter/environment writer already produces):

```properties
suite=Denali_Regression_v1
```

Every `mvn test` invocation that writes into the same `target/allure-results`
directory with the same `suite` value keeps merging into that one
ResideoNextGen report entry. Two things make this work:

- **Don't run `mvn clean` between the partial runs you want merged** —
  that's what deletes `target/allure-results` and would erase the earlier
  attempts. Each processing pass recomputes totals from *everything*
  currently in that directory, so as long as it isn't wiped, results keep
  accumulating under the same report.
- **Rerun-safe by design**: if a test fails on one attempt and passes on a
  later one (same Allure `historyId`), only the latest attempt counts —
  see `Normalizer.collapseRetries`. A test is never double-counted or left
  incorrectly marked as failed just because an earlier attempt failed.

**Starting a genuinely new regression cycle?** Change `suite` to a new
value (e.g. `Denali_Regression_v2`) **and** run `mvn clean test` first.
Skipping the clean means the new report would include every previously
accumulated result too, just relabeled — `ReportEngineIntegrationTest`
covers exactly this handoff (stage B merges under v1, stage C clears the
directory before writing v2's results and asserts two independent
entries).

(`resideonextgen.suite` also works if you'd rather avoid any chance of
colliding with a `suite` key some other tool writes to that same file.)

## What it reads, what it writes

Reads exactly what the Node processor reads (`../README.md` →
"What the processor reads"): `*-result.json`, `environment.properties`,
`executor.json`, `categories.json`. Writes the same canonical file set:
`summary.json`, `executions.json`, `trends.json`, `features.json`,
`tests.json`, `failures.json`, `categories.json`, `environment.json`, plus
the internal `.failures-contributions.json` bookkeeping file. A project
using this Java path and one using the Node/GitHub Actions path produce
dashboards the UI cannot tell apart.

## Configuration reference

All configurable via system properties (`-Dkey=value`) unless noted:

| Property | Default | Meaning |
|---|---|---|
| `resideonextgen.allureResultsDir` | `target/allure-results` | Where to read Allure results from |
| `resideonextgen.outputDir` | `target/resideonextgen` | Base dir; data is written to `<outputDir>/data` |
| `resideonextgen.historyLimit` | `50` | Executions retained in `executions.json`/`trends.json` |
| `resideonextgen.autoServe` (or env `RESIDEONEXTGEN_AUTO_SERVE`) | `false` | Serve immediately after the TestNG suite/Cucumber run finishes |
| `resideonextgen.port` (Mojo only) | `0` (any free port) | Preferred port for `mvn resideonextgen:serve` |
| `resideonextgen.skipProcessing` (Mojo only) | `false` | Serve existing data without reprocessing Allure results |
| `resideonextgen.openBrowser` | `true` | Open the dashboard URL in the default browser once the server starts (`mvn resideonextgen:serve` and auto-serve both honor this) |

## Known limitations

- Bundled dashboard assets are built once (from the current `../dist`
  build) and embedded as classpath resources at `resideonextgen-reporter-core`
  build time. If the frontend changes, rebuild `java-reporter` to pick up
  the new bundle: `npm run build` in `../`, then `mvn clean install` here.
- Multi-shard CI (results split across separate machines/`target/` dirs)
  isn't merged automatically — the `suite` grouping key above only
  accumulates results that land in the *same* `allure-results` directory.
- Structured device info (`EnvironmentInfo.device` in the frontend's
  model) is intentionally never populated by this module today — it's
  omitted from `environment.json` rather than fabricated. Add it once a
  capture path actually reports it (e.g. an Appium/ADB-sourced Allure
  label).

### Validated against a real run

Unlike the initial hand-written-but-unexecuted version, this module has
now actually been built and run end-to-end (`mvn test`, `mvn clean
install`, `mvn resideonextgen:serve`, real Selenium/Cucumber tests against
saucedemo.com), which surfaced and fixed two real bugs the code review
alone missed:

1. **Wrong Content-Type for the bundled JS module** — `index.html` loads
   its script as `<script type="module">`, and browsers refuse to execute
   a module script if the server's `Content-Type` header isn't a JS MIME
   type. `DashboardServer` no longer relies on the JDK's unreliable
   `URLConnection.guessContentTypeFromName`; content type is now resolved
   purely by file extension (`DashboardServerTest` locks this in).
2. **Output JSON didn't match the frontend's canonical model** — an
   earlier version let Gson serialize internal model classes directly,
   which silently produced the wrong shape (missing `comparison`/
   `generatedAt` entirely, `durationMs` instead of `duration`, uppercase
   enum values instead of lowercase, epoch-millis numbers instead of
   ISO-8601 date strings, `executions.json` shaped like the internal model
   instead of the frontend's leaner `RecentExecutionRow`). The missing
   `comparison` field crashed the whole Overview page right after its
   first render (KpiCardsRow reads `comparison.total.available`
   unconditionally). Fixed by adding an explicit mapping layer
   (`JsonViewBuilder`) that builds every JSON shape field-by-field against
   `src/types/models.ts`, plus a `ComparisonEngine` that computes real
   vs-previous-run deltas from execution history. `JsonViewBuilderTest` and
   `ReportEngineIntegrationTest` cover this directly.

If you're extending this module with a new output file, follow the same
pattern: build the shape explicitly in `JsonViewBuilder` against the
matching TypeScript interface, don't let Gson serialize an internal model
class directly.
